## Overview

The payment service is a Kubernetes-hosted REST API responsible for processing
card payments, managing refunds, and reconciling ledger entries. It integrates
with Stripe, PayPal, and an internal fraud-detection system, and is implemented
as a stateless Go service behind an API gateway.

An automated analysis of the architecture document identified seven findings
across the service's core components. Three findings are HIGH priority,
representing concrete risks to availability, data integrity, and security
that require immediate attention. The overall architecture is functional but
has significant reliability and security gaps that must be addressed before
the service can be considered production-ready for high-traffic or regulated
environments.

## Key Findings

### 1. Fraud Detection Has No Timeout or Circuit Breaker

**Impact**: HIGH

The payment processor calls the fraud-detection service synchronously on every
transaction above $500, with no timeout configured. Because the fraud service
has a p99 latency of 800 ms, a single slow or unresponsive fraud service
instance will stall the payment processor indefinitely — causing cascading
timeouts that affect all concurrent requests. In production this can render
the entire payment flow unavailable for minutes at a time.

### 2. Ledger Writes Have No Retry Logic — Payments Can Be Lost

**Impact**: HIGH

After a payment is successfully authorised by the card provider, the result is
written to the PostgreSQL ledger synchronously. If that write fails (database
overload, transient network error), the service returns a 500 error to the
caller — but the payment has already been charged. This creates a dangerous
inconsistency: the customer is billed, but the platform has no record of it.
There is no dead-letter queue or retry mechanism to recover these events.

### 3. API Key Validation Allows Potential Cache Poisoning

**Impact**: HIGH

The API gateway validates authentication keys by querying Redis without first
checking the key's format. A specially crafted key string could be used to
manipulate Redis cache lookups, potentially bypassing authentication or
polluting the cache with invalid entries. This is a security vulnerability at
the service boundary.
